## Why Most People Get Bad Results

Here's the deal. Most people install Claude Code, type a question, get a mid answer, and walk away thinking "AI isn't that useful yet."

But the problem isn't Claude Code. It's that they skipped the one file that makes it actually work.

Without CLAUDE.md, every session starts blind. Claude Code doesn't know your stack. Doesn't know your coding style. Doesn't know that your team uses Tailwind, not styled-components. Doesn't know you hate verbose output. It's guessing. And guessing produces mid results.

## OK So What Is It Actually?

CLAUDE.md is a markdown file at the root of your project that tells Claude Code how to behave. Think of it like onboarding a new team member. You wouldn't just hand someone a laptop and say "go write code." You'd tell them:

- What the project is and how it's structured
- What tech stack you use
- How you like things done
- What mistakes to avoid

That's literally what CLAUDE.md does. Except this team member reads your instructions perfectly, every single time, and never forgets them during the session.

## The Five Things Your CLAUDE.md Needs

Most people write one section and call it done. That gets you maybe 30% of the value. Here's the full picture:

### 1. Tell it what it's working with

```markdown
# CLAUDE.md

This is a Next.js 16 app using App Router, Tailwind CSS, and TypeScript.
Database: Neon Postgres via Drizzle ORM.
Auth: Clerk.
Deploy target: Vercel.
```

Simple. But you'd be shocked how many people skip this. Without it, Claude Code has to infer your stack from your `package.json` every single time. Sometimes it guesses wrong.

**Pro tip:** If you have a complex workspace, point to files instead of inlining everything. CLAUDE.md is an index, not a dump.

```markdown
Read `docs/architecture.md` for the full system design.
Read `docs/conventions.md` for coding standards.
```

### 2. Define how sessions start and end

This is the most underused part and honestly the biggest game-changer.

```markdown
## Session Lifecycle

**Starting**: Read MEMORY.md. Check handoffs/ for the latest handoff. Resume context.
**Ending**: Write a handoff if work is in-progress. Save any corrections to memory.
```

Without this, every session is stateless. You close the tab, everything's gone. With it, sessions compound. After a month, Claude Code knows your projects, your preferences, and your common patterns without you saying a word.

I know what you're thinking: "That sounds like extra work." It is, for about a week. Then it starts saving you 15-20 minutes per session. The math is not close.

### 3. Tell it how you communicate

```markdown
## How I Work

- Direct, no fluff. Skip preambles.
- NO em dashes. Use colons or split sentences.
- Lead with recommendations, not option lists.
- Code should be production-ready, not "here's a starting point."
```

This is huge and nobody talks about it. Without explicit preferences, Claude Code defaults to that verbose, hedging, "here are several approaches you might consider" mode. You know the one. It's polite but useless when you need a straight answer.

Put your communication preferences in CLAUDE.md and suddenly every response matches how you actually want to work.

### 4. Set the rules for how it works

```markdown
## Workflow Rules

1. Plan first for anything non-trivial. Think before coding.
2. One sub-agent per focused task. Keep the main chat clean.
3. After ANY correction from me, save it to memory. Don't make the same mistake twice.
4. Verify before calling it done. Run tests. Check the diff.
```

This is the difference between "doing things" and "doing things well." Without orchestration rules, Claude Code will just start writing code the moment you ask. Sometimes that's fine. For anything complex, you want it to plan first, break down the work, and verify the result.

### 5. Register your tools

```markdown
## Available Tools

Skills: `_context/skills/` (keyword-triggered workflows)
Agents: `_context/agents/` (specialist sub-agents)
Rules: `_context/rules/` (coding standards)
Commands: `.claude/commands/` (slash commands)
```

This tells Claude Code what it has to work with. Skills, agents, hooks, custom commands. If it doesn't know they exist, it won't use them.

## The Three Mistakes Everyone Makes

### Mistake 1: Dumping everything inline

Your CLAUDE.md is 2000 lines long. It has your entire coding style guide, all your project history, and your life story. Claude Code reads it all at the start of every session, eats a chunk of your context window, and starts ignoring half of it.

**Fix:** Keep CLAUDE.md under 100 lines. Use it as a table of contents that points to other files. Let Claude Code read those on demand.

### Mistake 2: No session lifecycle

You don't define start/end protocols. Every session starts cold. You spend 10 minutes re-explaining what happened yesterday.

**Fix:** Add the session lifecycle section above. Two minutes of investment at the end of each session saves ten minutes at the start of the next one. Every single time.

### Mistake 3: Being vague

"Write clean code" is not a rule. It's a wish. Claude Code reads that and goes "sure, clean code, got it" and then writes whatever it was going to write anyway.

**Fix:** Be specific. "Functions under 50 lines. Files under 800 lines. No mutation. Handle errors explicitly. Use early returns." Now Claude Code has actual constraints to follow.

## A Real CLAUDE.md in Production

This isn't theoretical. Here's a real workspace structure that runs daily across 6 projects:

```
CLAUDE.md (index, ~80 lines)
├── _context/
│   ├── skills/         (19 workflow skills)
│   ├── agents/         (9 specialist agents)
│   ├── rules/          (10 coding rules)
│   ├── handoffs/       (session continuity)
│   └── memory/         (persistent knowledge)
├── .claude/
│   ├── commands/       (8 slash commands)
│   └── hooks/          (automation triggers)
└── memory/
    ├── MEMORY.md       (index)
    ├── user_*.md       (who you are)
    ├── feedback_*.md   (corrections)
    ├── project_*.md    (active work)
    └── reference_*.md  (external pointers)
```

Every session starts in under 10 seconds. Zero re-explanation. Corrections from last month still apply. Handoffs from yesterday load automatically.

It took about a week to set up. It saves 30+ minutes daily. The ROI is absurd.

## Getting Started (5 Minutes)

You don't need all of this on day one. Start small:

**Right now:** Run `claude /init` in your project. It generates a basic CLAUDE.md from your codebase. That alone is a massive improvement over nothing.

**This week:** Add communication preferences and basic session lifecycle (read memory at start, write handoff at end).

**This month:** Add orchestration rules as you discover patterns. Add skills for repeated workflows.

The system grows with you. That's the point. You don't need to architect the whole thing upfront. You need to start, and then let it compound.
