## You're Wasting 10 Minutes Every Session

Real talk. If you open Claude Code and the first thing you do is explain what project you're working on, what happened yesterday, and what you need to do today, you're doing it wrong.

That's not an AI limitation. That's a workflow problem. And it's fixable in about five minutes of setup.

## The Three Phases

Every productive session follows the same pattern: start warm, work smart, end with a trail. Here's how.

### Phase 1: Start (Don't Start Cold)

When a session begins, Claude Code reads three things:

1. **Memory** (`MEMORY.md`): Who you are, what you're working on, your preferences, your past corrections
2. **Latest handoff**: What happened last session, what decisions were made, what's next
3. **Project docs**: Current state of your active projects

You configure this in CLAUDE.md:

```markdown
## Session Lifecycle

**Starting**: Read MEMORY.md. Check handoffs/ for latest handoff. Resume context.
```

That's it. Three reads. Zero re-explanation. It picks up exactly where you left off.

The difference is night and day. Instead of "let me explain the project again," you get "continue with the dashboard feature" and it just... knows.

### Phase 2: Work (Stop Guessing, Start Planning)

During a session, four rules make the difference between "using Claude Code" and "being productive with Claude Code":

**Plan before you build.** If the task has more than two steps, don't just start coding. Enter plan mode. Align on the approach. Then execute. This sounds slower. It saves tokens and prevents "that's not what I wanted" loops.

**Delegate to sub-agents.** Research in one agent, code review in another, test execution in a third. Your main conversation stays focused on decisions. Sub-agents do the grunt work.

**Capture every correction.** This is the one most people skip and it's the one with the highest ROI. When Claude Code makes a mistake and you fix it, that correction needs to get saved as a memory entry. Otherwise next session, same mistake. It's like training a team member who gets amnesia every night.

**Verify before done.** Nothing is "done" until tests pass, logs are clean, and the behavior matches what you asked for. "It looks right" is not verification.

### Phase 3: End (Two Minutes That Save Ten)

Before you close the session:

1. **Write a handoff** if work is in-progress. Doesn't need to be fancy. Three bullet points:
   - What was done
   - What's next
   - Any decisions or blockers

2. **Save corrections** if Claude Code made mistakes you had to fix. These become permanent rules.

Here's a handoff template:

```markdown
# Dashboard Feature - 2026-03-29

## What was done
- Layout component finished, responsive grid working
- Chart component 80% done, missing tooltip

## Key decisions
- Went with Recharts over Chart.js (better React integration)
- Using server components for the data layer

## What's next
- Finish chart tooltips
- Add loading skeleton
- Wire up real API data
```

That took 60 seconds to write. Tomorrow's session reads it and you save yourself 10 minutes of "where was I?" fumbling.

## The Memory Types (What to Save, What to Skip)

Not everything belongs in memory. Here's the cheat sheet:

| Save this | Skip this |
|-----------|-----------|
| Your role and preferences | Code patterns (read the code instead) |
| Corrections and confirmations | Git history (use `git log`) |
| Active project context | Debugging solutions (the fix is in the code) |
| Where to find external things | Temporary task details |

Four types of memory, each with a purpose:

- **user**: Who you are, how you work ("Senior PM, prefers direct communication")
- **feedback**: What you corrected ("Don't mock databases in tests. We got burned last quarter.")
- **project**: What's happening ("Auth rewrite is compliance-driven, not tech debt")
- **reference**: Where to look ("Pipeline bugs tracked in Linear project INGEST")

Each memory is its own markdown file. MEMORY.md is just an index that links to them.

## What This Actually Looks Like Over Time

| Timeframe | What changes |
|-----------|-------------|
| **Week 1** | Sessions start faster. Less re-explanation. Feels like extra work writing handoffs. |
| **Month 1** | Memory captures your style. Common mistakes stop happening. |
| **Month 3** | It knows your projects, people, and preferences. Anticipates what you need. |
| **Month 6** | Institutional knowledge is encoded. New team members could read your memory to onboard. |

This isn't a chatbot getting smarter. It's your practice getting more efficient. Small daily investments that compound over time. The people who set this up and stick with it for a month never go back.
